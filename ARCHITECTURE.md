# Архитектура проекта LinearCutWpf (Level 1)

## Общее описание
**LinearCutWpf** — это WPF-приложение (Windows Presentation Foundation), предназначенное для линейного раскроя материалов. Приложение использует паттерны проектирования MVVM (Model-View-ViewModel) и сервис-ориентированный подход для разделения логики, данных и пользовательского интерфейса.

## Структура модулей
Проект разделен на следующие ключевые модули (папки):

### 1. Models (Модели данных)
Содержит основные классы данных, используемые во всем приложении для хранения информации об артикулах, деталях для раскроя и доступных хлыстах (остатках).
- **Models.cs**: `ArticleItem`, `CutPart`, `StockItem`.
- **Docs:** `Docs/Classes/Models/Models.md`

### 2. ViewModels (Модели представления)
Реализуют паттерн MVVM для связывания пользовательского интерфейса (Views) с бизнес-логикой и моделями. Управляют состоянием и командами.
- **ViewModelBase.cs**: Базовый класс для всех ViewModels, реализующий `INotifyPropertyChanged`.
- **RelayCommand.cs**: Реализация `ICommand` для делегирования логики команд.
- **ManualCutViewModel.cs**: Модель представления для экрана ручного раскроя.
- **Docs:** `Docs/Classes/ViewModels/...`

### 3. Services (Бизнес-логика и сервисы)
Изолированные классы, выполняющие тяжелую бизнес-логику, такие как алгоритмы раскроя и управление настройками.
- **CuttingService.cs**: Основной сервис управления процессом раскроя. Координирует вызовы к `CutOptimizer` и возвращает `OptimizationResult`.
- **CutOptimizer.cs**: Алгоритм упаковки (1D Bin Packing / First Fit Decreasing) для оптимизации распределения деталей по хлыстам.
- **CutSettingsProvider.cs**: Управление настройками раскроя (например, толщина реза), паттерн Singleton.
- **ProfileHeightService.cs**: Сервис для сохранения и загрузки данных о видимой высоте профилей с метаданными (значения по умолчанию, ручные изменения).
- **Docs:** `Docs/Classes/Services/...`

### 4. Controls (Пользовательские элементы управления - Views)
Набор `UserControl` и `Window`, составляющих пользовательский интерфейс. Каждый контрол отвечает за свою часть функционала в главном окне.
- **ArticleSettingsControl.xaml.cs**: Управление списком артикулов (добавление, удаление).
- **DataSettingsControl.xaml.cs**: Ввод данных о деталях и хлыстах для выбранного артикула.
- **ArticleGroupingControl.xaml.cs**: Таблица общего списка деталей с группировкой по артикулам и подсветкой ошибок.
- **GroupingControl.xaml.cs**: Агрегирующий контрол, объединяющий настройки данных и группировку, а также управляющий валидацией перед раскроем.
- **ValidationDialog.xaml.cs**: Диалоговое окно отображения ошибок валидации.
- **ExportControl.xaml.cs**: Экспорт и импорт данных (Excel, CSV) и управление буфером обмена.
- **ManualCutControl.xaml.cs**: Экран ручного раскроя (связан с `ManualCutViewModel`).
- **ResultsControl.xaml.cs**: Отображение результатов оптимизации и статистики.
- **Docs:** `Docs/Classes/Controls/...`

### 5. Converters (Конвертеры данных для UI)
Классы, реализующие `IValueConverter` для преобразования данных при привязке в XAML.
- **BoolToColorConverter.cs**: Конвертирует `bool` в `Brush` (используется для подсветки строк).
- **HeightValueColorConverter.cs**: Конвертирует свойства высоты профилей в цвета фона для визуальной индикации.
- **Docs:** `Docs/Classes/Converters/...`

## Взаимодействие модулей (Flow)
1. **Загрузка/Ввод данных**: Пользователь взаимодействует с `ExportControl` для загрузки данных или вводит их вручную через `ArticleSettingsControl` и `DataSettingsControl`.
2. **Проверка (Валидация)**: В `GroupingControl` происходит агрегация данных и проверка на наличие ошибок (например, длина детали больше длины хлыста). Если есть ошибки, показывается `ValidationDialog`.
3. **Раскрой**: По команде пользователя данные передаются в `CuttingService`, который настраивается через `CutSettingsProvider` и вызывает `CutOptimizer` для генерации плана раскроя.
4. **Результат**: Сгенерированный `OptimizationResult` передается в `ResultsControl` для визуализации таблиц и статистики.
5. **Ручной режим**: Дополнительно доступен `ManualCutControl`, который работает через `ManualCutViewModel` для ручного вмешательства.

## Правила разработки
- **UI и Логика**: UI-события (в Code-Behind) должны быть минимальными и служить только для вызова сервисов или обновления View. Бизнес-логика выносится в Services или ViewModels.
- **Документация**: При добавлении нового класса или изменении существующих связей, необходимо обновлять `Level 2` документацию (файлы `.md` в папке `Docs/Classes/...`) и поддерживать актуальность данного `ARCHITECTURE.md`.
- **Комментарии**: Все публичные классы, свойства и методы должны содержать XML-комментарии (`/// <summary>`).
